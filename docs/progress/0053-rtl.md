# 0053 — RTL（返航）：一键收伞

**Commit**: `8a06910`  **Date**: 2026-04-22  **Milestone**: M6.3

## What / 做了什么

`MAV_CMD_NAV_RETURN_TO_LAUNCH`（id 20）——自动返航，连同 M6.2 的 home + M5.7 的 LAND 把一个完整的"无人机安全收尾"闭合：

3 阶段状态机：
1. **Climbing**：如果当前 z 比 home − `rtl_safe_alt_m` 高（即低于安全高度），先爬升；已经够高则直接跳 step 2。
2. **Returning**：保持 `rtl_safe_alt_m`，水平飞向 home xy。到达容差圈（默认 1 m）→ 跳 step 3。
3. **LandingState::Landing**：自动切到 LAND 模式，触地后 auto-disarm（M5.7 的逻辑复用，零新代码）。

```rust
pub enum RtlPhase { Idle, Climbing, Returning }
```

- `outer_step` 在 Climbing 态 override setpoint 成"垂直爬升到 safe_z，xy 保持不动"；
- 在 Returning 态 override 成"home xy + safe_z"（位置环把这俩融合成带俯仰的爬升飞行）；
- 转移在 outer_step 开头一次判断：`z ≤ safe_z + 0.3 → Returning`；`‖xy − home_xy‖ < 1 m → LandingState::Landing`；
- Home 没 latch（比如从没 arm 过）时 RTL 自动清零——避免"返航到 0,0" 的灾难。

Demo 层：Arc<Mutex<RtlPhase>> 共享；GCS 发 RTL → `Climbing`；`COMMAND_ACK` 收 20 → ACCEPTED。

**测试**：+3 app-copter（no-home clear / 完整 3 阶段 / 跳 climb）+1 sim-hil（RTL vs LAND 区分），26 app-copter + 25 sim-hil 测试全绿。

## Why / 为什么这么做

### 为什么 3 阶段不用 2 阶段

"直接飞到 home xy + safe_z 一步到位" 听起来简单，但是：
- 如果 vehicle 当前很低（比如 hover 在 -1m）、home 在远处（100 m）、safe_alt = 10 m，
- "一步到位" 控制器会给出一个斜向上的速度 vector，飞机以 ~45° 俯仰飞；
- 斜穿过可能的障碍物高度层（树、楼）——不安全；
- 位置环没有"先爬完再前进"的概念，它会同时做两个维度的减误差。

**拆 3 阶段 = 明确 "先升后飞后降"**。每个阶段只有一个主导轴的运动，安全、可预测、方便调试。

### 为什么 xy tolerance 1 m，不用 AltitudeReachedDetector 那种带时间的

Returning 的终点判定"我到家了"很简单：距离 < 1m。不需要时间积分：
- Landing 和 AltReached 用时间是因为要**排除短暂过零**（风扰吹过去又吹回来）；
- Returning 结束 → 切 Landing，Landing 会做自己的 1 秒触地时间积分——**两层保险**已经够；
- 简单的距离判定让 hand-off 快，不会在 home 上空绕圈。

### 为什么 Landing 时 RTL 会自动清零（通过"hand-off"）

把 `flight.rtl_phase = RtlPhase::Idle; flight.landing_state = LandingState::Landing;` 同时写，而不是留 RTL 在 Returning。这样：
- Landing override（M5.7）天然**dominant** — Returning 不会和 Landing 竞争 setpoint；
- RTL 状态"已完成" 语义清楚——不需要 GCS 手动发指令切换；
- Auto-disarm 后 `LandingState::Idle`、`RtlPhase::Idle`，飞机回到"默认 Disarmed 在地面"的起点。

### 为什么 no-home 时 clear 而不是 reject

发 RTL 但没有 home：
- **Reject**（PX4 的做法）：返回 `TEMPORARILY_REJECTED` / `DENIED`，飞机保持原状态——但状态机里标记过了 `Climbing`，得还原；
- **Clear 到 Idle**（我的选择）：不 ACK 拒绝，直接在 outer_step 里把 phase 归零——简单。对于 demo 来说 GCS 侧会看到 ACCEPTED（乐观 ACK）但实际无动作。

生产级应该走 reject + `COMMAND_ACK::DENIED + result_param2` 附加原因（比如"no home set, send DO_SET_HOME first"）。M6.4 会加 preflight reject。

### 为什么 safe_alt 默认 10 m

- 5 m：过树梢 / 单层建筑 borderline，不够余量；
- 10 m：双层建筑、标准通信塔下、避开大部分地面障碍；
- 30+ m：长距离 mission 返航，但 demo 场景（SITL 没障碍）不必要。

应当根据场景 set。`rtl_safe_alt_m` 是 `RateLoopConfig` 字段，运行时可变。

### 为什么不在 outer_step 里并行做 RTL phase 判定 + setpoint override

现在的结构：
```rust
// Phase transitions (mutate flight.rtl_phase)
match flight.rtl_phase { ... }

// Setpoint override based on CURRENT flight.rtl_phase
let active_sp = if ... { ... };
```

转移和覆盖**串行** — 先判是否该转到下个阶段，**然后** 用新阶段决定这一步的 setpoint。这样 Climbing → Returning 切换后**同一 tick 就用 Returning 的 setpoint**，不差一拍；更响应。

## Follow-ups / 遗留

- **Preflight reject**：无 home 时 ACK `MAV_RESULT_DENIED` 而不是 silent clear。
- **Obstacle-avoidance during transit**：Returning 阶段按直线飞，不避障。简单版：Agilicious-style waypoint 插值 + clearance buffer；高级版：DDS 接感知节点的 depth / costmap。
- **RTL from within Mission**：任务执行中如果信号丢失 / 电量低，应该自动触发 RTL（由 FDIR 模块决定）。目前只能手动 RTL。
- **RTL 时 yaw**：现在 yaw 没覆盖，用 caller setpoint 的 yaw；应该让 vehicle 面向飞行方向（Returning 时）或保持初始 heading。
- **`rtl_alt_final`**：有些飞控允许 RTL 到达 home 后悬停（不降），等操作员决定。我们直接 LAND——激进但简单。
- **Speed control in Returning**：现在 position controller 自己决定速度，可能太快撞停。应 velocity saturate 到 `rtl_transit_speed_mps`。
- **完整 SITL 闭环测试 RTL 全流程**：从 arm → takeoff → fly away → RTL → land 的**单一** SITL 测试。目前各段独立测。
- **RtlPhase Kani 证明**：状态转移单调性（Idle → Climbing → Returning → Idle）不能 bypass，Climbing 永远先于 Returning，等。
