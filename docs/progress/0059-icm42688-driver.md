# 0059 — ICM-42688-P IMU 驱动

**Commit**: `f371158`  **Date**: 2026-04-22  **Milestone**: M7.4

## What / 做了什么

给 Pixhawk 6X 主 IMU 写了个纯 `embedded-hal` SpiDevice 的驱动，放在 `core-hal::imu::icm42688`。

API：
```rust
let mut drv = Icm42688::new(spi)?;          // 查 WHO_AM_I=0x47
drv.configure()?;                            // 1 kHz / ±16 g / ±2000 dps
let sample = drv.read_sample(timestamp_us)?; // core_hal::ImuSample，SI 单位
```

数据流：
- Read framing: 首字节 `0x80 | addr`，后面字节装回响 data
- Write framing: 首字节 `0x7F & addr`，后面跟 value
- Burst read: `TEMP_DATA1 (0x1D)` 起连续 14 bytes = temp(2) + accel_xyz(6) + gyro_xyz(6)
- 所有 16-bit 数据 big-endian

标度常数来自 TDK 数据手册：
- accel ±16g → 2048 LSB/g → scale = `9.80665 / 2048` m/s²/LSB
- gyro ±2000dps → 16.4 LSB/dps → scale = `(π/180) / 16.4` rad/s/LSB
- temp = raw / 132.48 + 25 °C

6 新测试（+ core-hal 原 7 = 13），workspace 189 tests 全绿。thumbv7em 编译通过。

## Why / 为什么这么做

### 为什么纯 embedded-hal，不直接绑 embassy-stm32

**便携性**：`Icm42688<SPI: SpiDevice>` 可以接 embassy-stm32、rp2040-hal、nrf-hal、任何 HAL。将来有人在 Teensy 上跑、或在 STM32G4 上跑，驱动代码不用改一行。

**测试性**：单元测试用纯内存 `MockSpi`，不需要真硬件也不用 `qemu-system-arm`。CI 能跑完所有 register framing 逻辑。

**分层**：CLAUDE.md 强调"algo-* 不知道硅片"。把硅片细节关进 `core-hal`，向上只暴露 `ImuSample` trait → algo-ekf 不关心是 ICM-42688 还是 BMI088。

### 为什么 WHO_AM_I 一定要查

SPI 总线物理上可能接错：
- MOSI/MISO 线接反 → 所有 read 回 `0x00` 或 bus 默认值；
- CS 接错 → 同时激活多个 slave，返回乱码；
- 焊接冷焊 → 间歇性通信正常 / 错乱；
- 芯片被损坏 / 型号不对 (ICM-42605 vs -42688 vs -45686 WHO_AM_I 都不同)。

**`WHO_AM_I` mismatch** 是最早能在 `new()` 抓到这些的 check。错误里带返回的 byte 让 log 能定位是"无响应 (0x00/0xFF)" 还是"错型号 (例如 -42605 = 0x42)"。

### 为什么默认 ±16g 不是 ±4g

四旋翼机动加速度：
- hover: ~1 g
- normal banking: 1-2 g
- aggressive acrobatic / freestyle: 3-5 g
- crash impact / emergency arrest: 10-30 g

±4g 跑 race drone 都不够用；±16g 留余量给**崩溃诊断**（重要——crash 数据指导 FDIR 改进）。数值精度损失：16g/2048 = 7.8 mg LSB，比 IMU 本身噪声 (~1 mg rms) 还细，没损失。

### 为什么默认 ±2000dps 不是 ±1000dps

Anger rate:
- hover: ~0.01 rad/s
- normal maneuver: 1-3 rad/s
- acro roll: 10-30 rad/s (≈570-1720 dps)
- aggressive race: 35+ rad/s (≈2000+ dps)

Agilicious 论文里 quadrotor agile 动作推到 1500°/s。我们定位工业巡检没那么激进，但 ±1000dps 会在猛操纵时 saturate → gyro 输出 clamp 到极值 → EKF 认为飞机没转了 → 姿态估计崩。

±2000dps 安全门槛。精度 LSB：2000/32768 ≈ 0.06 dps LSB ≈ 1 mrad/s，远低于 IMU 噪声 floor。

### 为什么 1 kHz ODR 不是 4/8 kHz

- **CLAUDE.md**：控制内环 1 kHz → IMU 采样 ≥ 1 kHz 就够；
- **更高 ODR 需要 FIFO + DMA**：否则 SPI 读 delay 会漏样本；
- **FIFO + DMA 是独立大工程**，M7.5+ 才做；
- **Betaflight 类 FPV 用 8 kHz** 是因为控制器 pid-loop 跑 8 kHz —— 我们不是那种定位。

### 为什么 PWR_MGMT0 = 0x0F 不是 0x03（仅 accel）

`0x0F` = `0b0000_1111`：
- bits [1:0] = 11 → accelerometer Low-Noise mode
- bits [3:2] = 11 → gyro Low-Noise mode
- bits [7:4] reserved，全 0

都开才能同时输出 accel + gyro。Low-Noise > Low-Power 两个模式里 Low-Noise 噪声小、耗电多 —— 飞控场景电量够。

### 为什么 burst read 14 bytes 而不是 6 次独立 read

单次 CS 激活读一整块：
- CS 切换本身有开销（~1 µs）
- SPI 42 MHz: 14 bytes ≈ 2.7 µs vs 6× 独立 read = 6 × 3 µs = 18 µs
- **7× 速度**；
- 数据**一致性**：同一刻读出的 accel + gyro + temp 时间对齐（单次采样点），不是分散时刻的混合。

对 1 ms control loop 的预算 WCET 500 µs 这点时间很重要。

### 为什么 `read_sample` 把 `timestamp_us` 作为参数而不是自己 timestamp

驱动**不知道时间**。`embedded_hal::SpiDevice` 没有时间感知，我们也不希望它有。

时间由调用方（firmware）提供，能够来自：
- `embassy_time::Instant::now().as_micros()` —— 1 kHz 触发任务时抓
- 或 SPI CS 上升沿硬件 timestamp —— 硬件 latency 最准
- 或 IMU 的 data-ready IRQ 时间戳 —— 中间路径

驱动只负责读数。分离关注点。

### 为什么 MockSpi 手写，不引 `embedded-hal-mock`

`embedded-hal-mock` 是个 **250 KB crate**，CI 编译时间 + workspace dep 图膨胀。而 MockSpi 这里不到 50 行。

MockSpi 的设计：**就是个 256-byte register file**，遵守真正芯片的 read/write framing。配合 `set_block()` helper 在测试里预置"寄存器内容"，driver read_sample() 去读这块内存 → 验证 scaling 数学对 → 全程零外部依赖。

### 为什么 `#[cfg(test)]` 里用 `#[allow(clippy::indexing_slicing)]`

`regs[usize::from(reg)] = val` —— 数组下标是 `u8` 范围转 usize，永远不可能越界，但 clippy 的 `indexing_slicing` 不会静态分析这点。production 里我们严格用 `.get(i)`；tests 里可读性优先。workspace-wide `indexing_slicing = "deny"` 保持不变，单 module 级 allow。

## Follow-ups / 遗留

- **M7.5 把 driver 接进 firmware**：`embassy_stm32::spi::Spi::new(...)` + ExclusiveDevice（CS + bus 绑定）→ `Icm42688::new(...)` → `rate_loop_task` 每 tick `read_sample`。这一步之后 firmware 吃真 IMU，vehicle 在桌面上能动起来。
- **Data-ready IRQ**：ICM-42688 INT1 pin 拉低标识"新样本准备好"；用 embassy-stm32 exti 等 IRQ 比 polling 精确且省电。
- **FIFO mode + DMA**：更高 ODR（4kHz、8kHz）和 jitter-free 采样需要 FIFO。M7.6 候选。
- **Self-test**：bank 1/2 有自检寄存器，上电触发可以验加速度计/陀螺通道正常（出厂 factory trim）。
- **二/三冗余 IMU**：Pixhawk 6X 有 ICM-45686 + BMI088，都是类似 SPI 接口、不同寄存器。M7.x 加上两个驱动然后 `algo-fdir` 里 median-voting。
- **Accel anti-alias LPF 配置**：`ACCEL_CONFIG1` / `GYRO_CONFIG1` 里有 on-chip LPF cutoff 选项，默认 ~260 Hz，可能要按机型调。
- **ODR 变配置方法**：现在 `configure()` 硬编码 1kHz，加个 `Config { gyro_odr: Odr, ... }` 参数。
